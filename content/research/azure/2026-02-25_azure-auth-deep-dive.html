Azure Authentication Deep Dive

Understanding Identity, Sessions, and Auth Handoffs — From First Principles

Why this doc exists: Auth in Azure feels opaque because Microsoft wraps it in layers of abstraction. This guide strips it back to the raw HTTP flows — who talks to whom, what gets passed, and where the trust lives. Every diagram shows the actual requests happening between browser, Azure, and your app.

Part 1: What You Already Know (Traditional Auth)

Let's start from what you understand — a standard web app with login:

TRADITIONAL WEB APP AUTH ═══════════════════════════════════════════════════════════════ Browser Your Server Your Database │ │ │ │── GET /login ─────────────►│ │ │◄── Login form HTML ────────│ │ │ │ │ │── POST /login ────────────►│ │ │ {email, password} │── Check password ─────────►│ │ │◄── User found, match ──────│ │ │ │ │ │── Create session ──────────►│ │ │ session_id = "abc123" │ │ │ user_id = 42 │ │ │ │ │◄── Set-Cookie: session=abc123 │ │ │ │ │── GET /dashboard ─────────►│ │ │ Cookie: session=abc123 │── Lookup session ─────────►│ │ │◄── user_id=42 ─────────────│ │◄── Dashboard HTML ─────────│ (Blake Thomson) │ │ │ │ YOU OWN EVERYTHING: ✓ The login form ✓ The password check ✓ The session store ✓ The user database ✓ Session cookie management

This works fine. You control the entire chain. But it means you're responsible for:

  • Storing passwords securely (hashing, salting)
  • Session management (expiry, invalidation, storage)
  • Password reset flows
  • Multi-factor authentication
  • Preventing brute force attacks
  • Managing user accounts (create, disable, delete)

In Azure, Microsoft handles all of that. The trade-off: you lose visibility into the mechanics, which is why it feels opaque.

Part 2: How Azure Auth Actually Works (The Raw Flow)

Azure uses OAuth 2.0 + OpenID Connect — the same standard that "Sign in with Google" uses. The key difference from traditional auth: your app never sees the user's password. Microsoft handles login entirely, and sends your app a signed token proving who the user is.

The key insight: Your app never sees the password. Microsoft handles login (step ⑤), your app just receives a signed token (step ⑧) that proves "this person is blake@csmns.org and they work at Cedars-Sinai." From there, you create a session just like you normally would.

What's Inside the Tokens

The id_token Microsoft sends back is a JWT (JSON Web Token) — a Base64-encoded JSON string signed by Microsoft. Your app can decode it and trust the contents because Microsoft signed it with their private key.

# The id_token decodes to something like this:
{
  "aud": "YOUR_APP_CLIENT_ID",           # This token is for YOUR app
  "iss": "https://login.microsoftonline.com/{tenant}/v2.0",
  "iat": 1740000000,                      # Issued at (timestamp)
  "exp": 1740003600,                      # Expires (1 hour later)
  
  # THE USER INFO:
  "sub": "abc-123-def-456",               # Unique user ID in this tenant
  "oid": "abc-123-def-456",               # Object ID (same in all apps)
  "name": "Blake Thomson",                # Display name
  "preferred_username": "blake.thomson@csmns.org",
  "email": "blake.thomson@csmns.org",
  "tid": "YOUR_TENANT_ID",                # Which organization
  
  # ROLES (if configured):
  "roles": ["Admin", "DataAnalyst"]
}

This is your user object. You don't need a users table with passwords. You get all this from the token.

Part 3: The Three Ways to Do Auth in Azure

Azure gives you three approaches, from easiest to most control:

Approach 1: Easy Auth (Zero Code)

What: Azure runs an auth sidecar container alongside your app. All that OAuth flow above? Azure does it for you. Your app just reads HTTP headers.

How: Toggle "Authentication" on your Container App in the portal. Select Microsoft Entra ID. Done.

Best for: Internal dashboards, prototypes, apps where you just need "only @csmns.org people can access this."

Approach 2: MSAL Library (Your Code Handles Auth)

What: You implement the OAuth flow yourself using Microsoft's MSAL library. Same flow as the diagram above, but your code handles the redirects, token exchange, and session management.

How: Install msal Python library, configure with your App Registration credentials, handle login/callback routes.

Best for: Apps where you need fine control — custom login pages, role-based access, token caching, calling other APIs on behalf of the user.

# FastAPI with MSAL — you control the flow
from msal import ConfidentialClientApplication
import os

msal_app = ConfidentialClientApplication(
    client_id=os.environ["AZURE_CLIENT_ID"],
    client_credential=os.environ["AZURE_CLIENT_SECRET"],
    authority=f"https://login.microsoftonline.com/{os.environ['AZURE_TENANT_ID']}"
)

@app.get("/login")
async def login():
    """Step ③ — Redirect user to Microsoft login"""
    auth_url = msal_app.get_authorization_request_url(
        scopes=["User.Read"],
        redirect_uri="https://yourapp.com/callback"
    )
    return RedirectResponse(auth_url)

@app.get("/callback")
async def callback(request: Request):
    """Step ⑦⑧ — Exchange code for tokens"""
    code = request.query_params.get("code")
    
    result = msal_app.acquire_token_by_authorization_code(
        code=code,
        scopes=["User.Read"],
        redirect_uri="https://yourapp.com/callback"
    )
    
    if "error" in result:
        return {"error": result["error_description"]}
    
    # Step ⑨ — Extract user info from the id_token
    user = result.get("id_token_claims")
    # user = {
    #   "name": "Blake Thomson",
    #   "preferred_username": "blake.thomson@csmns.org", 
    #   "oid": "abc-123-def-456",
    #   "tid": "your-tenant-id"
    # }
    
    # Create YOUR session (just like traditional auth)
    session_id = create_session(user)
    response = RedirectResponse("/dashboard")
    response.set_cookie("session", session_id)
    return response

@app.get("/dashboard")
async def dashboard(request: Request):
    """Step ⑩ — Normal session-based access"""
    session = get_session(request.cookies.get("session"))
    if not session:
        return RedirectResponse("/login")
    
    return {"message": f"Welcome {session['name']}", "data": get_data(session['oid'])}

Approach 3: Bot Framework Auth (Teams Bots)

What: Teams bots get user identity automatically from the Bot Framework. The JWT is validated by the SDK — you just read the activity object.

How: The Bot Framework SDK handles everything. The user's identity comes from their Teams login — no extra auth flow needed.

Best for: Teams bots (obviously). The user is already logged into Teams with their @csmns.org account.

TEAMS BOT AUTH FLOW ═══════════════════════════════════════════════════════════════ User (Teams) Microsoft Bot Service Your Container App │ │ │ │ User is ALREADY │ │ │ logged into Teams │ │ │ (blake@csmns.org) │ │ │ │ │ │── "Show me Q4 data" ───►│ │ │ │ Bot Service adds │ │ │ Microsoft-signed JWT │ │ │ to the request │ │ │ │ │ │── POST /api/messages ───►│ │ │ Authorization: Bearer │ │ │ eyJ... (signed JWT) │ │ │ │ │ │ Body: { │ │ │ type: "message", │ │ │ text: "Show me Q4", │ │ │ from: { │ │ │ id: "29:...", │ │ │ name: "Blake", │ │ │ aadObjectId: │ │ │ "abc-123-..." │ ◄── SAME ID as Easy Auth! │ │ } │ │ │ } │ │ │ │ │ │ Bot Framework SDK │ │ │ automatically validates│ │ │ the JWT signature │ │ │ │ │ │◄── Response ─────────────│ │◄── Bot reply ───────────│ │ YOUR CODE: user_name = turn_context.activity.from_property.name # "Blake Thomson" user_aad_id = turn_context.activity.from_property.aad_object_id # "abc-123-..." # SAME oid as what you'd get from Easy Auth or MSAL

The magic: aadObjectId is the same across ALL three approaches. Whether a user hits your dashboard (Easy Auth), calls your API (MSAL), or chats with your bot (Bot Framework) — the oid / aadObjectId is always abc-123-def-456. This is how identity carries across apps.

Part 4: Restricting Access — "Only My Team"

You asked: "How do I make a Teams bot only accessible by my team?" There are multiple layers of restriction:

Layer 1: Tenant Restriction (Org-Wide)

When you create an App Registration as Single Tenant, only users in your Azure AD tenant (e.g., @csmns.org) can authenticate. Anyone with a @gmail.com or @otherhospital.org account is blocked automatically.

# In your bot's BotFrameworkAdapterSettings:
_settings = BotFrameworkAdapterSettings(
    app_id=os.environ["MICROSOFT_APP_ID"],
    app_password=os.environ["MICROSOFT_APP_PASSWORD"],
    channel_auth_tenant=os.environ["MICROSOFT_APP_TENANT_ID"],  # ← THIS LINE
    # Only users from THIS tenant can talk to the bot
)

This is the minimum. Every internal tool should use Single Tenant.

Layer 2: User Assignment (Specific People)

To restrict beyond "everyone in our org" to "only these specific people":

1
Go to Entra ID → Enterprise Applications → find your app
2
Properties → set "User assignment required" to Yes
3
Users and groups → Add user/group → add only your team members

Now only those specific people can access the app. Everyone else in the org gets an "AADSTS50105: Your administrator has not given you access" error.

Layer 3: App Roles (Permissions Within Your App)

For role-based access (Admin vs. Viewer vs. Editor):

1
App Registration → App roles → Create app role
Example: DataAnalyst, Admin, Viewer
2
Enterprise Application → Users and groups → assign roles to users
3
Roles appear in the JWT token → your code checks them
# The token now includes roles:
{
  "name": "Blake Thomson",
  "preferred_username": "blake.thomson@csmns.org",
  "roles": ["Admin", "DataAnalyst"]    # ← These come from Entra
}

# In your FastAPI app:
@app.get("/admin/settings")
async def admin_settings(request: Request):
    token = decode_token(request)
    if "Admin" not in token.get("roles", []):
        raise HTTPException(403, "Admin access required")
    return get_admin_settings()

Now the practical question: "How do I build an app where users log in, have sessions, and their edits are tracked?"

# Your CRUD endpoints — auth is handled by the sidecar
from fastapi import Request, HTTPException
from datetime import datetime

@app.post("/api/providers/{npi}/edit")
async def edit_provider(npi: str, request: Request):
    # Auth already done by Easy Auth sidecar
    user_email = request.headers.get("X-MS-CLIENT-PRINCIPAL-NAME")
    user_id = request.headers.get("X-MS-CLIENT-PRINCIPAL-ID")
    
    if not user_email:
        raise HTTPException(401, "Not authenticated")
    
    body = await request.json()
    
    # Write the edit with full audit trail
    db.execute("""
        INSERT INTO providers.edits (npi, field_name, new_value, edited_by, edited_at)
        VALUES (%s, %s, %s, %s, %s)
    """, (npi, body["field"], body["value"], user_email, datetime.utcnow()))
    
    return {"status": "saved", "edited_by": user_email}

@app.get("/api/providers/{npi}/history")
async def edit_history(npi: str, request: Request):
    """See who edited what and when"""
    user_email = request.headers.get("X-MS-CLIENT-PRINCIPAL-NAME")
    
    rows = db.execute("""
        SELECT field_name, new_value, edited_by, edited_at
        FROM providers.edits
        WHERE npi = %s
        ORDER BY edited_at DESC
    """, (npi,))
    
    return {"npi": npi, "edits": rows, "requested_by": user_email}

Notice what you DON'T need: No users table. No password hashing. No session store. No login page. No "forgot password" flow. Entra ID handles all of that. Your database only stores what users did (edits, activities), not who users are.

# Dashboard forwarding user's token to the API:
import httpx

@app.get("/analytics")
async def analytics(request: Request):
    # Easy Auth gives us the user's access token
    user_token = request.headers.get("X-MS-TOKEN-AAD-ACCESS-TOKEN")
    
    # Forward it to the API service
    async with httpx.AsyncClient() as client:

            "https://bd-api-service.azurecontainerapps.io/api/analytics",
            headers={"Authorization": f"Bearer {user_token}"}
        )

@app.get("/api/analytics")

Part 7: The App Registration — What It Actually Is

This is the piece that confuses people the most. An App Registration is your app's identity card in Azure. Let's make it concrete:

Think of it like this:

You (a person) have an ID card (driver's license). You show it to prove who you are.

Your app also needs an ID card. The App Registration IS that ID card. It tells Microsoft: "This application exists, it belongs to this organization, and here's what it's allowed to do."

Part 9: Decision Tree — Which Approach to Use

START HERE │ ├─► "I just need to block unauthorized access" │ └─► Easy Auth (zero code, toggle in portal) │ ├─► "I need to know WHO the user is for audit/personalization" │ └─► Easy Auth + read X-MS-CLIENT-PRINCIPAL headers │ ├─► "I need role-based access (admin vs viewer)" │ └─► Easy Auth + App Roles in Entra + check roles in code │ ├─► "I need custom login flow or UI" │ └─► MSAL library (you control the OAuth flow) │ ├─► "I'm building a Teams bot" │ └─► Bot Framework SDK (auth is automatic) │ └─► Restrict with: Single Tenant + User Assignment + in-code checks │ ├─► "My dashboard needs to call my API as the user" │ └─► Forward the access token (on-behalf-of flow) │ └─► "I need to restrict to specific people, not just the org" └─► Enterprise Application → User Assignment Required = Yes → Add specific users/groups

The bottom line: Azure auth feels complex because Microsoft abstracted away the parts you used to own (passwords, sessions, user DB). But the trade is good — you get enterprise-grade auth (MFA, SSO, conditional access, audit logs) for free. Your job shrinks to: (1) register your app, (2) choose Easy Auth or MSAL, (3) read the user identity from headers/tokens, (4) use that identity in your business logic.
