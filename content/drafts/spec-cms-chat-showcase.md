### Example SQL (generated by LLM, never exposed to user)

```sql
-- "Top cardiologists in LA by Medicare volume"
SELECT
    provider_name,
    provider_city,
    SUM(total_medicare_payment) AS total_payments,
    COUNT(DISTINCT hcpcs_code) AS unique_services
FROM cms_utilization
WHERE provider_state = 'CA'
  AND provider_city LIKE 'LOS ANGELES%'
  AND specialty = 'Cardiology'
GROUP BY provider_name, provider_city
ORDER BY total_payments DESC
LIMIT 20;
```

```sql
-- "Total Medicare payments by state"
SELECT
    provider_state AS state,
    SUM(total_medicare_payment) AS total_payments,
    COUNT(DISTINCT npi) AS provider_count
FROM cms_utilization
GROUP BY provider_state
ORDER BY total_payments DESC;
```

### Frontend

- **Framework:** React + TypeScript + TailwindCSS
- **Layout:** Split pane — chat sidebar (30%) | chart canvas (70%)
- **Chat:** Scrollable message history, user/assistant bubbles, input with send button
- **Charts:** Render in main canvas, NOT inline in chat
- **Chart library:** Recharts (bar, line, pie, scatter, area) + react-simple-maps (choropleth)
- **Chart types supported:** bar, horizontal bar, grouped bar, line, area, pie, scatter, choropleth map, stat cards, data table
- **Responsive:** On mobile, chat and chart stack vertically

### Backend

**Endpoint:** `POST /api/cms/query`

```json
// Request
{
  "query": "Show me top cardiologists in LA by Medicare volume",
  "session_id": "optional-for-context"
}

// Response
{
  "chartType": "horizontal_bar",
  "title": "Top Cardiologists in Los Angeles by Medicare Payment Volume",
  "summary": "The top 20 cardiologists in LA processed $142M in Medicare claims...",
  "data": [
    { "name": "Dr. Smith", "total_payments": 8420000, "claim_count": 12400 },
    { "name": "Dr. Jones", "total_payments": 7180000, "claim_count": 10800 }
  ],
  "metadata": {
    "row_count": 20,
    "query_time_ms": 340,
    "filters_applied": { "state": "CA", "city": "LOS ANGELES", "specialty": "Cardiology" }
  },
  "axes": {
    "x": { "key": "total_payments", "label": "Total Medicare Payments ($)", "format": "currency" },
    "y": { "key": "name", "label": "Provider" }
  }
}
```

**LLM Query Pipeline:**
1. User query → Claude API (extract intent, filters, aggregation, chart type)
2. LLM returns structured JSON: `{ tables, columns, filters, groupBy, orderBy, limit, chartType }`
3. SQL builder constructs query from structured output (NOT raw LLM SQL)
4. Execute against DuckDB read-only connection
5. Format response with chart metadata

### Data Layer

- **Database:** DuckDB, file `provider_searcher.duckdb` on Hetzner VPS
- **Connection:** Read-only, single concurrent connection with query queue
- **Pre-defined views** for performance:
  - `v_provider_summary` — aggregated provider stats
  - `v_state_payments` — state-level payment rollups
  - `v_specialty_metrics` — specialty performance
  - `v_prescribing_patterns` — drug category aggregations
- **Row limit:** 10,000 rows max per response
- **No PII:** All data is publicly available CMS data (no patient info)

## 4. Security & Rate Limiting

| Control | Implementation |
|---------|---------------|
| Auth | None (public demo) |
| Rate limit | 10 queries/hour/IP via Redis counter |
| Query timeout | 30 seconds max (DuckDB pragma) |
| SQL injection | No raw SQL from LLM — structured builder with parameterized queries |
| Table allowlist | Only permitted tables/views can be queried |
| Column allowlist | Only permitted columns per table |
| Write protection | Read-only DuckDB connection (`access_mode = 'read_only'`) |
| Response size | Max 10,000 rows, truncated with warning |
| Input sanitization | Query string max 500 chars, stripped of special chars |

### SQL Builder Allowlist

```python
ALLOWED_TABLES = {
    "cms_utilization": ["npi", "provider_name", "provider_city", "provider_state",
                        "specialty", "hcpcs_code", "total_medicare_payment",
                        "total_services", "total_beneficiaries"],
    "cms_prescriber": ["npi", "provider_name", "specialty", "drug_name",
                       "generic_name", "total_claims", "total_cost"],
    "cms_quality":    ["npi", "provider_name", "specialty", "mips_score",
                       "quality_score", "pi_score"],
}

ALLOWED_AGGREGATIONS = ["SUM", "COUNT", "AVG", "MIN", "MAX", "COUNT DISTINCT"]
```

## 5. User Experience

### Loading States
- Typing indicator in chat while query processes
- Skeleton chart placeholder in canvas
- Progress text: "Analyzing query..." → "Running against 90M rows..." → "Rendering chart..."

### Error Handling
- **Query too complex:** "I can handle simpler queries — try asking about one topic at a time."
- **Timeout:** "That query was too broad. Try narrowing by state or specialty."
- **Rate limit:** "You've used your 10 free queries this hour. Come back soon, or contact Blake for unlimited access."
- **No results:** "No data matched that query. Try different filters."

### Quick-Start Buttons
Pre-built example queries displayed below the input:
- 🏥 "Top cardiologists in LA"
- 💊 "Prescribing patterns: CA vs TX"
- 📊 "MIPS scores by specialty"
- 🗺️ "Medicare payments by state"
- 🏗️ "Facility type distribution"

### Chart Interactions
- **Hover:** Tooltip with exact values
- **Click bar/slice:** Filter to that segment (drill-down)
- **Download:** PNG screenshot or CSV data export
- **Share:** URL with encoded query parameter (`?q=base64query`)
