WebMCP: Making Websites AI-Native

Technical Overview & Implementation Guide — February 2026

What this document covers: WebMCP is a browser API proposal (backed by Google & Microsoft) that lets websites expose their functionality as MCP tools directly to AI agents in the browser. This guide explains what it is, how it differs from standard MCP, the use cases it enables, and how to implement it — written as both a technical reference and an agent-ready prompt for building WebMCP into web applications.

01 The Problem WebMCP Solves

Today, when an AI agent needs to interact with a website, it has two options — both bad:

🤖 Browser Automation (Playwright, Puppeteer)

The agent takes screenshots, parses HTML, clicks buttons, fills forms — trying to use the site like a human would.

  • Token-expensive (full HTML/screenshots in context)
  • Non-deterministic (page layouts change, elements move)
  • Fragile (breaks when UI updates)
  • Slow (render → screenshot → parse → decide → act)

🔌 Separate MCP Server

The developer builds a standalone MCP server that exposes API endpoints. The AI calls the API directly, bypassing the website entirely.

  • Requires building & maintaining a separate system
  • Disconnected from the UI (AI can't see what user sees)
  • Every website would need its own MCP server
  • Doesn't scale to millions of websites

The core tension: Websites are built for humans. AI agents struggle to use them. Building a separate MCP server for every site doesn't scale. Browser automation is unreliable and expensive.

WebMCP's answer: Let the website itself declare what an AI agent can do with it. The website becomes its own MCP server — tools are registered per page, contextually, with strict schemas. No screenshots, no HTML parsing, no guesswork.

02 What WebMCP Actually Is

WebMCP is a browser-native API (currently in Chrome Canary behind a flag) that adds a new capability to the browser: navigator.modelContext. This API allows web page JavaScript to:

  1. Register tools that AI agents can discover and call
  2. Unregister tools when navigating away from a page or unmounting a component
  3. Define input/output schemas so the AI knows exactly what parameters to pass
  4. Execute JavaScript functions when a tool is called — updating React state, making API calls, navigating, whatever you need

The tools are contextual per page. When the user navigates from a search page to a detail page, the available tools change automatically. This is the key innovation — the AI's capabilities adapt to what the user is currently looking at.

Two Modes of Declaration

📝 Declarative (HTML Attributes)

For simple form-filling use cases. Add attributes directly to HTML form elements:

  • tool-name — Names the tool
  • tool-description — Tells the AI when to use it
  • tool-param-description — Describes each input field

The browser automatically converts the form into an MCP tool. Zero JavaScript required.

Best for: static sites, contact forms, booking forms, simple interactions

⚡ Imperative (JavaScript API)

For dynamic applications. Register tools programmatically via navigator.modelContext.registerTool():

  • Full control over tool definition
  • Execute arbitrary JavaScript on tool call
  • Access React/Vue/Svelte state
  • Register/unregister on component mount/unmount

Best for: SPAs, React apps, dynamic dashboards, any complex interaction

03 WebMCP vs. Standard MCP vs. Tambo

These three technologies serve different purposes and are complementary, not competing:

Dimension WebMCP (Browser API) Standard MCP Server Tambo (Generative UI)
Where it runs In the browser, on the web page Backend server (local or remote) React app with hosted/self-hosted backend
What the AI interacts with The live website UI & state APIs, databases, services Component library with streaming props
Who provides the AI? Browser-native (Gemini in Chrome, Copilot in Edge) or extensions External client (Claude Desktop, ChatGPT, etc.) Built-in agent loop (your API key)
Tools are... Contextual per page — change as user navigates Static — all tools available at connection time Component schemas — AI picks which to render
Best for AI operating your existing website AI accessing your data from anywhere AI generating UI inside a chat
User experience User stays on your site, AI helps navigate & act User is in the AI's interface, data comes from your backend User is in a chat, components render inline

The insight: WebMCP makes your website AI-accessible. Standard MCP makes your data AI-accessible. Tambo makes your UI AI-generatable. A complete platform might use all three — WebMCP on the dashboard, standard MCP for headless access, Tambo for chat-embedded experiences.

04 The Contextual Tool Pattern

This is the most powerful concept in WebMCP and the thing that distinguishes it from every other approach.

In standard MCP, when an AI connects to your server, it gets all your tools at once. 50 tools? All in context. Most irrelevant to the current task. Token-expensive and noisy.

In WebMCP, tools exist only in the context where they make sense. They register when a component mounts and unregister when it unmounts:

// Search Page Component
useEffect(() => {
  const tool = navigator.modelContext.registerTool({
    name: "search_providers",
    description: "Search healthcare providers by specialty and location",
    inputSchema: { specialty: { type: "string" }, location: { type: "string" } },
    execute: (params) => {
      setFilters(params);    // Update React state
      navigateToResults();   // Trigger navigation
    }
  });
  return () => tool.unregister();  // Clean up on unmount
}, []);

// Results Page Component — DIFFERENT tools available
useEffect(() => {
  const tools = [
    navigator.modelContext.registerTool({
      name: "list_results",
      description: "Get all providers currently shown with details",
      execute: () => providers  // Returns current React state as JSON
    }),
    navigator.modelContext.registerTool({
      name: "set_filter",
      description: "Filter current results by criteria",
      inputSchema: { insurance: { type: "string" }, maxDistance: { type: "number" } },
      execute: (params) => applyFilters(params)
    }),
    navigator.modelContext.registerTool({
      name: "log_outreach",
      description: "Log an outreach call to a provider",
      inputSchema: { npi: { type: "string" }, outcome: { type: "string" }, notes: { type: "string" } },
      execute: (params) => { api.logCall(params); refreshCard(params.npi); }
    })
  ];
  return () => tools.forEach(t => t.unregister());
}, [providers]);

The AI's available tools adapt in real-time as the user navigates. On the search page: search tools. On the results page: filter and action tools. On a provider profile: detail and CRM tools. The context window stays lean and relevant.

05 Use Cases: Healthcare Data Platform

Here's how WebMCP applies to a bespoke healthcare analytics platform deployed in a client's Azure environment:

🔍 Scenario 1: AI-Assisted Provider Search

A physician liaison opens the provider dashboard in Chrome. They ask the browser's AI:

"Find cardiologists in West LA who accept Medicare with high referral volume"

User speaks to browser AI
→ AI discovers search_providers tool on current page
→ AI calls tool with {specialty: "cardiology", location: "West LA", insurance: "Medicare"}
→ Your React app updates state → page re-renders with filtered results
→ AI discovers new tools: list_results, sort_by, set_filter
→ AI calls list_results → reads structured provider data from React state
→ AI recommends top providers based on volume data
Result: Dashboard navigated, filtered, and analyzed — no screenshots, no HTML parsing

📞 Scenario 2: CRM Updates From the Dashboard

The liaison just finished a call. While looking at Dr. Smith's profile on the dashboard:

"Log an outreach call — connected with Dr. Smith, she's interested in joining the network. Schedule a follow-up for next Thursday."

User is on provider profile page
→ Available tools: log_outreach, get_provider_details, schedule_followup
→ AI calls log_outreach with {npi: "1234567890", outcome: "connected", notes: "Interested in joining network"}
→ Your app writes to CRM database, updates the activity timeline on screen
→ AI calls schedule_followup with {npi: "1234567890", date: "2026-03-05"}
→ Calendar event created, follow-up badge appears on provider card
Result: CRM updated, follow-up scheduled, UI reflects changes — all from natural language

📊 Scenario 3: AI Reads the Dashboard for You

An executive opens the analytics dashboard and asks:

"Summarize what I'm looking at and create a PowerPoint for my board meeting"

User is on analytics overview page
→ Available tools: get_dashboard_data, get_chart_data, get_kpi_summary
→ AI calls get_dashboard_data → receives clean JSON of all metrics, charts, KPIs
→ AI uses its OWN capabilities (Claude's deck generation, Gemini's formatting) to build the presentation
→ AI has perfect structured data — not a screenshot interpretation
Result: The AI leverages your data + its own strengths. You provide the intelligence, it provides the output format.

🌐 Scenario 4: Multi-Page Workflow

A data analyst needs to compare two markets:

"Compare our cardiology market share in West LA vs. San Fernando Valley, then show me the top 5 providers we're losing referrals to in each area"

AI calls search_providers with West LA params → page navigates
→ AI calls get_market_share → reads West LA metrics from page state
→ AI calls search_providers with San Fernando params → page navigates again
→ AI calls get_market_share → reads SFV metrics
→ AI calls get_referral_leakage → structured competitor data
→ AI compiles comparison, presents findings in natural language
Result: Multi-step analysis across pages, all deterministic, all using your tools

07 Implementation Guide

Step 1: Feature Detection

// Check if WebMCP is available in the browser
const hasWebMCP = 'modelContext' in navigator;

// Your app works for humans regardless — WebMCP is progressive enhancement
if (hasWebMCP) {
  registerPageTools();
}

Step 2: Create a React Hook for Tool Registration

// hooks/useWebMCPTool.ts
import { useEffect } from 'react';

interface WebMCPToolDef {
  name: string;
  description: string;
  inputSchema?: Record<string, any>;
  execute: (params: any) => any;
}

export function useWebMCPTool(tool: WebMCPToolDef, deps: any[] = []) {
  useEffect(() => {
    if (!('modelContext' in navigator)) return;

    const registered = (navigator as any).modelContext.registerTool({
      name: tool.name,
      description: tool.description,
      inputSchema: tool.inputSchema,
      execute: tool.execute,
    });

    return () => registered.unregister();
  }, deps);
}

// hooks/useWebMCPTools.ts — register multiple tools at once
export function useWebMCPTools(tools: WebMCPToolDef[], deps: any[] = []) {
  useEffect(() => {
    if (!('modelContext' in navigator)) return;

    const registered = tools.map(t =>
      (navigator as any).modelContext.registerTool(t)
    );

    return () => registered.forEach(r => r.unregister());
  }, deps);
}

Step 3: Register Tools Per Page/Component

// pages/ProviderSearch.tsx
function ProviderSearch() {
  const [results, setResults] = useState([]);
  const [filters, setFilters] = useState({});

  useWebMCPTool({
    name: "search_providers",
    description: "Search for healthcare providers. Returns matching providers "
      + "with name, NPI, specialty, location, and contact info.",
    inputSchema: {
      specialty: { type: "string", description: "Medical specialty (e.g. cardiology, orthopedics)" },
      location: { type: "string", description: "City, zip code, or region name" },
      insurance: { type: "string", description: "Insurance type accepted (optional)" },
    },
    execute: async (params) => {
      const data = await api.searchProviders(params);
      setResults(data.providers);
      setFilters(params);
      return { count: data.providers.length, providers: data.providers };
    }
  }, []);

  useWebMCPTool({
    name: "list_current_results",
    description: "Get all providers currently displayed on screen with full details",
    execute: () => ({
      filters: filters,
      count: results.length,
      providers: results.map(p => ({
        name: p.name, npi: p.npi, specialty: p.specialty,
        address: p.address, phone: p.phone,
        medicare_volume: p.claims_count, quality_score: p.quality
      }))
    })
  }, [results, filters]);

  // ... normal React rendering for human users
}

08 Strategic Implications

For Developers Building Bespoke Applications

  • Build one app that serves two audiences. The same React dashboard works for humans clicking buttons AND for AI agents calling tools. No separate bot or integration needed.
  • Don't compete with foundation models. Claude, Gemini, and GPT are spending billions on reasoning capabilities. Your value is the data, domain expertise, and tool definitions. Let them be the brain.
  • Progressive enhancement. WebMCP is additive. Your app works perfectly for humans without it. When the browser supports it, AI capabilities appear automatically.
  • Future-proof against AI churn. Today it's Gemini in Chrome. Tomorrow it might be something else. WebMCP is a standard — your tool definitions work with any compliant AI.

For the Business Model

  • WebMCP-enabled apps are inherently more valuable — they're accessible to both humans and AI agents, multiplying the utility of every feature you build.
  • Reduced support burden. Instead of building help docs and training materials, the AI can guide users through the app using the tool definitions you've already written.
  • Compounding intelligence. As browser AIs improve (better reasoning, longer context, multimodal understanding), your WebMCP-enabled tools automatically benefit. Your app gets smarter without you shipping updates.
  • Lock-in through tooling, not data. The client's data stays in their environment. Your value is the tool architecture, domain schemas, and data enrichment — things they can't easily replicate.

09 Current Status & Limitations

As of February 2026: WebMCP is a proposal, not a shipped standard. It's available in Chrome Canary behind the chrome://flags/#web-mcp flag. There's also a Chrome extension (MCPB) that polyfills the API for current browsers.

What Works Today

  • Chrome Canary with flag enabled
  • Both declarative (HTML attributes) and imperative (JavaScript API) modes
  • Tool registration, unregistration, and execution
  • Input schemas with type validation
  • MCPB Chrome extension for polyfill support

Limitations & Open Questions

  • Security: Poisoned tool descriptions could mislead AI agents. How much control should a page have over the AI's behavior?
  • Browser support: Currently Chrome-only. Adoption depends on other browsers implementing the standard.
  • AI integration: The browser-native AI (Gemini, Copilot) needs to be the one consuming these tools. Third-party AI apps (Claude Desktop) would need browser extension bridges.
  • Auth & permissions: How do tools handle authenticated actions? The current model assumes the browser session handles auth.
  • Trust model: Should the AI auto-execute tools, or always confirm with the user? Chrome's current approach includes a confirmation UI.

10 Agent Prompt: Building WebMCP Into a Web Application

The following prompt can be used to guide a coding agent (Claude Code, Cursor, etc.) when adding WebMCP support to an existing React application:

TASK: Add WebMCP (browser Model Context Protocol) support to this React application. WebMCP lets the web page register MCP tools that browser-based AI agents can discover and call.

WHAT WEBMCP IS:
- A browser API accessed via navigator.modelContext.registerTool()
- Tools are registered per page/component and unregistered on unmount
- Each tool has: name, description, inputSchema (optional), execute function
- The execute function runs client-side JavaScript — it can update React state, call APIs, navigate, etc.
- Tools are contextual: different pages expose different tools

IMPLEMENTATION RULES:
1. Create a useWebMCPTool(toolDef, deps) hook that feature-detects navigator.modelContext and handles register/unregister on mount/unmount
2. Create a useWebMCPTools(toolDefs[], deps) hook for registering multiple tools
3. WebMCP is progressive enhancement — the app MUST work without it. Always check for feature support.
4. Register tools at the PAGE level, not the app level. Each route/page should register only the tools relevant to what the user is currently viewing.
5. Tool descriptions should be clear, specific, and include what the tool returns — the AI uses these to decide when/how to call tools.
6. Input schemas should use descriptive field descriptions with examples where helpful.
7. Execute functions should return structured data (JSON) that's useful for the AI, not raw React state.
8. For forms that should be AI-fillable, use declarative mode: add tool-name, tool-description, and tool-param-description attributes to form/input elements.

TOOL CATEGORIES TO CONSIDER PER PAGE:
- Read tools: Get current page data in structured format (e.g., list_results, get_dashboard_data)
- Navigate tools: Search, filter, sort, paginate (e.g., search_providers, set_filter)
- Action tools: Create/update records, trigger workflows (e.g., log_call, schedule_followup)
- Export tools: Get data in formats the AI can use for external tasks (e.g., export_chart_data)

EXAMPLE TOOL REGISTRATION:
useWebMCPTool({ name: "search_items", description: "Search for items by keyword and category. Returns array of {id, name, price, category}.", inputSchema: { query: { type: "string", description: "Search keyword" }, category: { type: "string", description: "Category filter (optional)" } }, execute: async (params) => { const results = await api.search(params); setResults(results); return { count: results.length, items: results }; } }, []);
